# 住所写像論 未検証項目まとめ

作成日: 2026-06-06

## 目的

このメモは、住所写像論、AGID/AOID、住所検証エンジン、ZK proof 関連のうち、現時点でまだ十分に検証済みとは言えない項目を整理するためのものです。

ここでの「未検証」は、次のいずれかを意味します。

- Lean 形式証明では扱えない実世界データの正しさ。
- GIS、郵便、配送、自然地理、商用 API 比較など、広域ベンチマークがまだ不足しているもの。
- 実装テストはあるが、暗号回路監査、外部監査、運用監査、規模検証がまだないもの。
- 論文では提案・設計・仮説として書けるが、断定表現にすると危険なもの。

なお、`formal/AMTCore.lean` は `lake env lean formal/AMTCore.lean` でコンパイル済みです。したがって、同ファイルで証明されている抽象定理は、このメモでは「未検証」ではなく「形式検証済み。ただし実世界への適用条件が必要」と扱います。

未検証項目を実行可能な予想へ変換するための台帳は、`docs/address-morphism-executable-expectations.md` に分離しています。この台帳では、各予想に実行方法、成功時文言、失敗時の書き換えを対応させます。未検証項目は、放置するものではなく、実験・監査・反例・適用条件へ変換する対象です。

## 形式検証済みだが、実世界では追加検証が必要なもの

| 項目 | Lean上の状態 | まだ必要な検証 |
| --- | --- | --- |
| 非単射観測下の完全解決不能性 | 形式検証済み | 実データでどの観測衝突が多いかを国別・用途別に測る。 |
| 候補集合に含まれない対象は解決できない | 形式検証済み | 候補生成の実世界 recall を測る。 |
| ambiguous / unresolved は非発行状態 | 形式検証済み | UI、API、監査ログで誤って PID を発行しないことを継続テストする。 |
| Certified gated resolution | 形式検証済み | 本番スコア、品質、鮮度、リスク値が抽象ゲートに正しく対応しているかを監査する。 |
| 正規化衝突・2D投影衝突・観測ベースPID衝突 | 形式検証済み | 多言語データ、縦方向住所、建物内区画で実例を集める。 |
| split は単一関数では表せない | 形式検証済み | 行政変更、分筆、建物分割、地名再利用の実データで履歴グラフを検証する。 |
| 公開述語だけでは秘密住所を一意に決めない条件 | 形式検証済み | 実際の匿名集合サイズ、属性の絞り込み過ぎ、ZK回路安全性を検証する。 |

## 最優先の未検証項目

| 優先度 | 未検証項目 | 何が不足しているか | 論文での安全な扱い |
| --- | --- | --- | --- |
| S | 世界規模の候補生成 completeness | 全世界・全住所体系・自然地理・建物内区画で真の対象が候補集合に入る保証はまだない。 | 「候補生成の十分性を仮定する」「coverage は地域別に検証する」と書く。 |
| S | 多言語検索の recall 向上 | 日本語、英語、ラテン文字、非ラテン文字、旧地名、別名、方言の大規模 benchmark が未完了。 | 「多言語展開は recall layer」と書き、同一性保存定理とは書かない。 |
| S | 自然地理・文化地物の世界的 coverage | 川、滝、湖、島、砂漠、湿地、氷河、洞窟、谷、遺跡、世界遺産などを全世界で認識する検証はまだない。 | 「対応可能な地物型を拡張する」と書き、「全世界の全名称を認識済み」とは書かない。 |
| S | GIS strict validation | 現状は hard error 0 だが警告が残る。警告ゼロの strict global validation ではない。 | 「機械可読性と hard error 0 は確認済み。strict は未完了」と書く。 |
| S | 住所検証エンジンの商用API比較 | Loqate、Experian、Melissa、Smarty と同一条件の広域比較が未実施。 | 「競合比較計画」または「評価条件別比較」と書き、勝利宣言は避ける。 |
| S | 本物のZK回路としての安全性 | TypeScript envelope と互換性テストはあるが、回路実装・証明システム選定・暗号監査が未完了。 | 「ZK-ready」「proof envelope」と書き、「完全なZKシステム」とは書かない。 |
| S | AGID/AOID の本番セキュリティ | threat model、鍵管理、失効、同期、QR、権限委譲、公開/非公開境界の外部監査が未完了。 | AMT本体から分け、AGID/AOID応用論文で扱う。 |

## 住所検証エンジンで未検証のもの

| 項目 | 未検証の理由 | 必要な検証 |
| --- | --- | --- |
| 何か国に本当に対応できるか | address format registry と公式ソース分類は進んでいるが、国別の実住所検証精度は別問題。 | 国別 gold dataset、郵便番号 lookup、行政区画照合、配送可否 benchmark。 |
| 郵便番号 API の全世界導入 | 公式 API がない国、登録必須、利用規約制限、商用制限、レート制限がある。 | 国・郵便局ごとの API availability、license、rate limit、terms audit。 |
| no-normal-postcode 地域 | 郵便番号が標準的でない地域では、郵便番号検証だけでは品質を保証できない。 | 地理境界、行政階層、地名、配送事業者データとの組み合わせ検証。 |
| 住所タブ品質スコアの calibration | 都市、田舎、島、山地、砂漠、湿地、氷原などで最適閾値が異なる。 | 地域タイプ別の false positive / false negative 測定。 |
| ユーザー非表示スコアの妥当性 | ユーザーにスコアを見せない方針はUX上良いが、内部判定の説明責任が必要。 | 注意表示、非表示、再検証対象化の監査ログと回帰テスト。 |
| 配送会社 API の登録なし利用 | 多くの配送会社 API は登録・契約・認証が必要。登録なしで使える範囲は限定的。 | 会社別 terms、公開 endpoint、rate limit、住所検証用途の可否確認。 |

## 自然地理・地図地物で未検証のもの

| 項目 | 未検証の理由 | 必要な検証 |
| --- | --- | --- |
| 世界全ての島名認識 | 島名データは言語差、別名、無人島、行政所属、境界曖昧性が大きい。 | OSM、GeoNames、Natural Earth、国別公式地名台帳の coverage 比較。 |
| 川・滝・湖・池 | 線、点、面、季節変化、名称重複、上流下流の範囲差がある。 | feature type 別の geometry 検証、代表点、流路、流域、行政所属の検証。 |
| 砂漠・荒野・草原・森林・湿地・氷原・氷河 | 境界が連続的・季節的・観測依存で、住所的参照と地形的範囲が一致しない。 | polygon quality、季節変化、保護区境界、地名レベルの benchmark。 |
| 洞窟・谷・山・橋・道路 | 点・線・面・標高・入口・通行可能性など、必要な参照粒度が違う。 | 地物型ごとの addressability schema を作り、検索・表示・近傍住所を検証。 |
| 遺跡・世界遺産 | UNESCO範囲、国内文化財範囲、観光名、現地名が一致しない場合がある。 | 公式 heritage source、OSM、国別文化財台帳との照合。 |
| 南極・基地情報 | 基地の運用状態、季節基地、国別運営、位置、名称変更、公開可否が変わる。 | COMNAP、SCAR、各国南極機関、OSM/Natural Earth の source audit。 |
| NASA / 宇宙機構データ | NASA open data は利用可能なものが多いが、住所検証に直接使えるとは限らない。 | データ用途、ライセンス、解像度、地名との接続、住所検証への寄与を検証。 |

## ZK proof 関連で未検証のもの

| 項目 | 未検証の理由 | 必要な検証 |
| --- | --- | --- |
| ZK Address Proof | 住所を秘匿したまま地域属性を証明する envelope はあるが、本物の回路証明は未監査。 | Circom/Noir/Rust等で回路化、soundness、zero-knowledge、witness leakage 監査。 |
| ZK Residence Proof | 「同一住所居住者」「指定国/都市居住者」の匿名集合が小さいと秘匿性が弱い。 | anonymity-set 下限、属性粒度制御、k-anonymity風の公開制限。 |
| ZK Delivery Eligibility | 配送可能性だけを証明しても、配送時には最終的な配送先開示が必要になる場合がある。 | 開示段階設計、配送業者への最小開示、replay対策、scope制御。 |
| nullifier による重複登録防止 | TS実装レベルの検証はあっても、回路内 nullifier 安全性と domain separation は未監査。 | domain separated nullifier、epoch、scope、unlinkability の暗号監査。 |
| 失効・鮮度証明 | root anchoring は設計できるが、発行者運用、時刻同期、失効遅延の検証が必要。 | revocation root、freshness window、issuer SLA、chain anchoring 実験。 |
| proof bundle registry 互換性 | 互換性テストはあるが、全 proof 間の衝突・再利用・追跡可能性は運用設計が必要。 | proof type、scope、challenge、audience、domain separation の仕様化。 |
| 匿名レート制限 | 不正利用防止と匿名性のバランスが未検証。 | rate limit epoch、nullifier reuse、DoS耐性、誤BAN対策。 |
| 品質しきい値証明 | 品質スコアを秘匿したまま閾値以上と証明するには、score生成過程の検証が必要。 | score commitment、source commitment、閾値回路、score操作耐性。 |

## 商用・運用・法務で未検証のもの

| 項目 | 未検証の理由 | 必要な検証 |
| --- | --- | --- |
| Loqate/Experian/Melissa/Smarty に負けない品質 | これらは proprietary data、配送事業者連携、国別正規化ノウハウを持つ。 | 同一データセット、国別、用途別、費用別、可用性別の比較実験。 |
| 公式ソースの全導入 | 公式データでも license、更新頻度、利用条件、商用可否が異なる。 | source registry に license、allowedUse、freshness、coverage を記録。 |
| 人道支援・災害住所 | 被災者・難民向け住所は有用だが、誤配・なりすまし・保護対象情報漏洩の危険が高い。 | NGO/行政運用設計、最小開示、失効、緊急時 governance。 |
| Address Reputation | 配送成功率などで信頼性推定は可能だが、バイアス・攻撃・差別リスクがある。 | 攻撃耐性、差別防止、説明可能性、データ保持ポリシー。 |
| DAOによる住所ルール管理 | 国・行政・郵便制度の正式性をDAOが代替できるわけではない。 | DAOは補助的提案・監査・翻訳規則管理に限定する設計。 |

## 論文で避けるべき断定表現

| 避ける表現 | 安全な表現 |
| --- | --- |
| 全ての住所を一意に解決できる。 | 条件付きで解決し、条件不足では ambiguous / unresolved を返す。 |
| 世界中の自然地理名を認識できる。 | 対応地物型を拡張し、coverage は source と GIS benchmark で地域別に検証する。 |
| 商用住所検証APIに勝てる。 | 同一条件で比較可能な benchmark を設計し、国別・用途別に評価する。 |
| ZK proof は完成している。 | 現状は proof envelope / ZK-ready 実装であり、完全なZK回路は別途実装・監査する。 |
| GIS検証済みなので地理的に完全に正しい。 | 現在のGIS検証は hard error 0、strict は警告が残るため未完了。 |
| 郵便局APIを全て導入済み。 | 公式ソース分類は進んでいるが、国別APIの availability と利用規約検証が必要。 |
| 住所スコアは住所の正しさを証明する。 | 品質スコアは内部のリスク制御信号であり、正しさの単独証明ではない。 |

## 次に検証すべき順番

1. GIS warning 149件を分類し、strict validation に近づける。
2. 候補生成 recall benchmark を都市、田舎、島、山地、砂漠、極地、自然地理で作る。
3. 多言語検索 benchmark を日本語、英語、ラテン文字、非ラテン文字、旧地名、別名で作る。
4. 住所タブ品質スコアの calibration dataset を作る。
5. 郵便番号・住所検証エンジンの国別 gold dataset を作る。
6. 公式郵便・行政・地名ソースの license / freshness / allowedUse を監査する。
7. PID発行 gate の本番スコアと Lean の抽象ゲートの対応を文書化する。
8. ZK proof は回路言語を選び、少数の proof から監査可能な形で実装する。
9. 商用API比較は、勝敗ではなく評価条件別の性能表として設計する。

## 結論

現時点で強く主張できるのは、住所写像論が「無条件完全解決器」ではなく、「候補生成、構造比較、履歴、品質、鮮度、リスク、監査、保留状態を持つ条件付き住所参照フレームワーク」であることです。

未検証の中心は、数学ではなく実世界 coverage、データ品質、暗号回路、商用比較、運用ガバナンスです。論文では、これらを「今後の検証対象」「応用論文で扱う対象」「実装仕様で扱う対象」として明示すると、過剰主張を避けながら研究の強さを保てます。
