# 住所写像論 実行可能な予想と失敗時の書き換え台帳

作成日: 2026-06-06

## 目的

住所写像論では、魅力的な予想や期待をそのまま断定すると危険です。一方で、予想を避けすぎると研究として弱くなります。

そこで、各予想を次の形に変換します。

```text
予想
-> 実行可能な検証
-> 成功時に書ける主張
-> 失敗時に書き換える主張
-> 実装または論文での扱い
```

この方式にすると、期待が外れても論文は弱くなりません。むしろ、住所写像論が扱うべき境界条件、反例、unresolved、安全ゲート、データ品質の説明が増えます。

## 基本方針

| 原則 | 内容 |
| --- | --- |
| 予想を実験可能にする | 予想には必ず dataset、command、metric、failure mode を付ける。 |
| 成功条件を事前に決める | 成功後に都合よく基準を変えない。 |
| 失敗を記述に変える | 失敗した場合は「理論が間違い」ではなく、適用条件、限界、反例、追加ゲートとして書く。 |
| 無条件主張を避ける | 全世界、全住所、全言語、全API、完全ZKなどの表現は検証前に使わない。 |
| 実装と論文を同期する | 実装で未対応なら、論文では「実装可能性」または「今後の検証」として扱う。 |

## 実行可能な予想台帳

| No. | 予想 | 実行方法 | 成功時に書けること | 期待が外れた時の新しい記述 |
| --- | --- | --- | --- | --- |
| 1 | 候補生成を強化すれば真の対象が候補集合に入りやすくなる。 | 国別・地域別 gold dataset を作り、`recall@k`、候補欠落率、unresolved率を測る。 | AMTの候補生成は、対象地域で実測 recall を改善した。 | 候補生成の十分性は地域・出典依存であり、欠落時は unresolved または追加証拠要求が必要である。 |
| 2 | 多言語展開は住所検索 recall を上げる。 | 日本語、英語、ラテン文字、非ラテン文字、旧地名、別名で検索 benchmark を作る。 | 多言語展開は、同一性判定の前段として recall を改善する。 | 多言語展開は同一性を保証しない。地域別 language pack と衝突検出が必要である。 |
| 3 | クラスタリングは表記揺れより安定する。 | 同一実体の表記違い、翻訳、略称、旧住所をクラスタ維持率で評価する。 | 参照保存的な表記変更では、同値類が表面表記より安定する。 | split、merge、地名再利用では同値類自体を更新する必要がある。 |
| 4 | unresolved は誤発行を減らす。 | PID発行前後の false issuance、near tie、low quality、stale source ケースを replay する。 | unresolved は失敗ではなく、誤った精密性を抑制する安全状態である。 | unresolved が多すぎる地域では、候補生成、出典、閾値、UI再確認を改善する必要がある。 |
| 5 | PID発行ゲートは監査可能性を高める。 | 候補生成、クラスタ、unresolved、履歴更新、PID発行を audit envelope で再検証する。 | PIDは、宣言されたゲートを通過したことを後から検査できる。 | ゲート通過は実世界正しさの証明ではない。ゲート条件と出典品質の監査が必要である。 |
| 6 | 住所履歴はグラフとして扱う方が関数より強い。 | 行政変更、分筆、建物分割、統合、改称データで split/merge を抽出する。 | Address Lineage は一対多、多対一、改称、廃止を表現できる。 | 記録が欠落する場合、履歴保存則は「記録された状態」に限定される。 |
| 7 | 住所エントロピーは識別対象数と粒度に応じて増える。 | 都市、田舎、島、山地、砂漠、建物内区画で候補数と必要識別ビットを測る。 | 住所情報量は、対象密度、粒度、文脈要求と関係する。 | 都市化そのものが原因ではない。グリッド住所や制度設計により短い住所でも衝突を減らせる。 |
| 8 | 文脈ごとに最適住所は変わる。 | 配送、消防、行政、不動産、災害支援で utility function を変えて top candidate を比較する。 | 住所解決の最適性は文脈相対的である。 | 文脈が同じ効用を持つ場合は同じ出力でよい。相対性は万能な主観性ではなく、効用差がある時に現れる。 |
| 9 | 自然地理名を住所的参照にできる。 | 川、湖、滝、島、砂漠、湿地、氷原、洞窟、谷、遺跡、世界遺産で feature benchmark を作る。 | AMTは、郵便住所が弱い地域でも自然・文化地物を参照対象にできる。 | 全世界の全地物名を認識済みとは言えない。source-bound な拡張として記述する。 |
| 10 | 住所タブ品質スコアは内部制御に使える。 | 都市、田舎、島、山地、砂漠、湿地、極地で false positive / false negative を測る。 | 品質スコアは、非表示、注意表示、再検証対象化の内部信号として使える。 | 品質スコアは真理ではない。低品質地域ではスコア表示ではなく追加確認や出典要求に変換する。 |
| 11 | 公式郵便・行政ソースを増やせば住所検証は強くなる。 | source registry に authority、license、freshness、coverage、allowedUse を入れ、国別評価する。 | 公式ソースは、国別住所検証の信頼性を高める。 | 公式ソースがない、古い、商用利用不可の地域では、global fallback と unresolved policy が必要である。 |
| 12 | 商用住所検証APIに対して競争できる。 | Loqate、Experian、Melissa、Smarty と同条件の国別・用途別 benchmark を作る。 | AMTは、透明性、監査可能性、自然地理、履歴、保留状態で差別化できる。 | 全面勝利ではなく、AMTは商用APIを置換するより、公開証拠・監査・保留判断を補完する層である。 |
| 13 | ZK Address / Residence / Delivery は住所秘匿に有効である。 | 回路実装、witness leakage、anonymity set、nullifier、scope、revocation を監査する。 | AMTの属性生成は、ZK住所証明の前段理論として使える。 | envelopeだけでは完全ZKとは言えない。回路実装・監査までは「ZK-ready」と書く。 |
| 14 | AGID/AOIDは買い物AgentやMCPに接続できる。 | API conformance、OpenAPI、MCP tool、Agent request/response、最小開示をテストする。 | AMT応用識別子は、Agentが配送可能性や住所参照を扱う基盤になる。 | Agent連携は応用仕様であり、AMT本体の正しさとは分離する。 |

## 論文本文に入れる短い説明

次の段落を「検証と限界」または「今後の課題」の冒頭に置くとよい。

```text
本稿の経験的主張は、予想として放置せず、実行可能な検証単位へ分解する。各予想には、入力データ、評価指標、成功基準、失敗時の書き換え規則を対応させる。予想が成功した場合は、AMTの適用可能範囲を拡張する。予想が失敗した場合は、失敗を反例、適用条件、unresolved条件、追加ゲート、出典不足、または応用論文への分離として記述する。したがって、AMTは検証結果に応じて主張を強化または縮退できる、反証可能な住所参照理論である。
```

## 期待が外れた場合の書き換えテンプレート

### 1. coverage が足りない場合

```text
本実験では、対象地域において候補生成の coverage が十分でないことが確認された。したがって、本地域では AMT の解決結果を verified として発行せず、unresolved または additional-evidence-required として扱う。これは理論の失敗ではなく、候補生成十分性仮定が満たされない場合の安全側挙動である。
```

### 2. 多言語検索で誤結合が出た場合

```text
多言語展開は recall を改善するが、同一性を保存する写像ではない。実験で観測された誤結合は、正規化衝突の実例である。したがって、多言語候補は identity decision ではなく candidate expansion として扱い、構造距離、出典、履歴、文脈ゲートによる二次判定を必要とする。
```

### 3. 自然地理の認識が不安定な場合

```text
自然地理名の addressability は、利用可能な地図ソース、地物種別、幾何形状、季節変化、行政所属に依存する。本実験で不安定だった地物型は、全世界対応ではなく source-bound referent として扱う。
```

### 4. 商用API比較で負けた場合

```text
商用住所検証APIは、一部地域において非公開データ、配送事業者連携、国別正規化ノウハウにより高い精度を示した。AMTの貢献は、それらを無条件に置換することではなく、証拠、履歴、保留、監査可能なPID発行、自然地理、垂直参照を統合する公開可能な形式層を提供する点にある。
```

### 5. ZK proof がまだ完成しない場合

```text
現段階の実装は、住所属性と監査エンベロープを秘匿証明へ接続するための proof-ready model である。完全なZK証明と呼ぶには、回路実装、証明システム、trusted setupまたはtransparent setup、nullifier設計、失効、鮮度、匿名集合評価、外部監査が必要である。
```

## 実装側の工夫

| 工夫 | 目的 |
| --- | --- |
| `expectationId` を検証ログに付ける | どの予想を検証したか追跡する。 |
| `expectedOutcome` と `observedOutcome` を分ける | 期待と結果を混同しない。 |
| `claimStatus` を持つ | `proved`, `tested`, `empirical-pass`, `empirical-fail`, `open`, `rejected` で管理する。 |
| failure rewrite を必須にする | 失敗時に論文の文章を自動的に安全側へ更新できる。 |
| source coverage と score calibration を分ける | データがあることと、スコアが正しいことを混同しない。 |
| unresolved率を成功指標に含める | 無理な解決より保留が正しいケースを評価できる。 |

## 推奨データ構造

```ts
type ClaimStatus =
  | 'proved'
  | 'tested'
  | 'empirical-pass'
  | 'empirical-fail'
  | 'open'
  | 'rejected';

type ExecutableExpectation = {
  id: string;
  claim: string;
  validationMethod: 'lean' | 'gis' | 'implementation-test' | 'benchmark' | 'security-audit' | 'governance-review';
  metric: string;
  passCriterion: string;
  expectedOutcome: string;
  observedOutcome?: string;
  status: ClaimStatus;
  successWording: string;
  failureWording: string;
  paperLocation: string;
};
```

## 結論

住所写像論の予想は、期待として置くだけでなく、実行可能な検証単位へ落とすべきです。期待が当たれば適用範囲を広げる。期待が外れれば、反例、条件、不可能性、unresolved、安全ゲート、別論文への分離として書く。

この方針により、AMTは「大きな主張をするが検証不能な理論」ではなく、「予想を実行し、失敗も理論の境界として吸収する研究プログラム」になります。
