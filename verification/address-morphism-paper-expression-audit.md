# 住所写像論PDF 表現・説明・誤値チェック

対象PDF: `C:/Users/kitau/Downloads/main (5).pdf`

## 総評

現行PDFは、住所写像論の中心構造をかなり丁寧に説明できている。特に以下は強い。

- 住所を文字列・座標・配送先から分離し、実体参照として扱っている。
- 「常に解ける」ではなく、候補不足・曖昧性・未解決出力を安全側として扱っている。
- 候補生成、クラスタ、履歴、確率、PID、可換図式、自然地理、垂直参照まで一貫した流れがある。
- 第14章でLean・実装検証・GIS検証を分離しており、誇張を避ける姿勢がある。

ただし、論文としてさらに強くするには、表現の強すぎる箇所、二重番号、数値例の導出条件、参照文献の表記を修正した方がよい。

## 表現を弱めた方がよい言い回し

### 1. 「住所は識別子として機能していない」

強すぎる。住所は多くの制度内では識別子として機能している。

推奨:

```text
現行の住所表記は、多くの局所制度では実用的な識別子として機能する。
しかし、国際的・多言語的・時間変化を含む機械可読な永続識別子としては、
同一性・再利用性・監査可能性が十分ではない。
```

### 2. 「配送は最も厳密なテストである」

配送は強い実務検証だが、唯一・絶対ではない。

推奨:

```text
配送結果は、住所参照の妥当性を検証する代表的かつ実務的な信号である。
ただし配送失敗は、住所参照の誤りだけでなく、不在、アクセス制約、天候、
配送主体のミスなど複数の原因を含むため、失敗理由を分解して扱う必要がある。
```

### 3. 「完全性」

現行PDFは条件付き完全性を丁寧に書いているが、章タイトルや説明に「完全」が多いため、読者が万能性と誤読しやすい。

推奨:

```text
条件付き完全性
手続き的完全性
モデル内部の一意性
観測条件付き健全性
```

のように、必ず修飾語を付ける。

### 4. 「真実へ収束する」

統計モデルとしては強い。

推奨:

```text
識別可能性と観測モデルの仮定が成立する場合、事後分布はモデル上の真の候補へ集中する。
```

## 数式・モデル上の要注意点

### エネルギー更新式

現行の式:

```text
E_{t+1}(c) = E_t(c) - T_t log P(e_{t+1} | c)
```

`0 < P <= 1` なら `log P <= 0` なので、右辺の増分 `-T_t log P` は非負である。したがって「成功はエネルギーを下げる」とは直接言えない。

安全な説明:

```text
高い尤度を持つ候補は低い尤度を持つ候補よりエネルギー増分が小さい。
そのため反復更新後、正しい候補は競合候補に比べて相対的に低エネルギーになる。
```

もし本当に「下げる」と書きたいなら、別式にする:

```text
E_{t+1}(c) = E_t(c) - eta * s(e_{t+1}, c)
```

ここで `s` は成功証拠の正の支持スコア。

### 構造非類似度 `D`

付録Bの例はよい。`D(a,b)=0.1539`、`D(b,a)=0.0939` は非対称性、`D(x,z)=0.7650 > D(x,y)+D(y,z)=0.7242` は三角不等式違反を示している。

ただし本文では、読者の混乱を避けるため、`距離` と呼ぶ箇所を減らし、基本名を `構造非類似度` または `方向付きコスト` に寄せた方がよい。

### 自然地理多様体

地表を滑らかな2次元多様体として近似する説明は理論モデルとして使える。ただし湿地、氷河、河道、海岸線、砂漠境界、行政境界は不連続・季節変動・観測依存を含む。

推奨:

```text
地形多様体は、局所的な解析と候補生成のための近似モデルであり、
制度境界・季節変動・災害変形・観測欠落を完全に表現するものではない。
```

### ホモトピー不変性とRPID不変性

地形変化に対してRPIDが常に不変、とは強すぎる。河川の付け替え、湖の干上がり、氷河後退、島の沈降、境界線変更では参照実体が分裂・統合・消滅しうる。

推奨:

```text
参照クラス保存的な連続変形に限り、RPIDは不変である。
分裂、統合、消滅、名称再利用が発生する場合は、Address Lineage上の遷移として扱う。
```

WO2014170646A1 / related patent family, priority 2013.

## 説明不足または追加した方がよい箇所

### 1. 図式の凡例

第12章の可換図式は強いが、初見読者には記号が多い。第12章冒頭に凡例表を置くと読みやすい。

追加例:

| 記号 | 意味 |
|---|---|
| `u` | 入力住所表現 |
| `cxt` | 用途・地域・言語・配送などの文脈 |
| `Phi_t` | 候補生成写像 |
| `kappa_t` | クラスタ/商写像 |
| `E_t` | 評価関数 |
| `S_t` | 選択/保留判定 |
| `psi_t` | PID発行写像 |

### 2. 例をもう少し実務寄りにする

数理章の後に短い例を挟むと理解が大きく上がる。

- 同一住所表記だが別実体: 同じビルの1階店舗と2階事務所。
- 同一実体だが別表記: 日本語住所、英語住所、略称、旧住所。
- 自然地理: 湖、川、滝、山頂、砂漠キャンプ、南極基地。
- 垂直参照: 地下街、空港ゲート、屋上施設、宅配ロッカー。

### 3. 第14章に「検証できないもの」を明記する

現行でも書いているが、さらに表にすると強い。

| 検証手段 | 検証できる | 検証できない |
|---|---|---|
| Lean | 抽象定理、反例、条件付き含意 | 実世界データの正しさ |
| GIS | 境界、包含、座標整合性 | 住所制度・社会名の完全性 |
| 実装テスト | アルゴリズム仕様の再現 | 世界中のデータ品質 |
| 運用監査 | 更新履歴、証拠root、PID発行過程 | 人間社会の全ての真実 |
