# 住所写像論 検証境界まとめ

作成日: 2026-06-06

## 目的

この文書は、住所写像論（AMT）、AGID/AOID、住所検証エンジン、ZK住所証明に関する主張を、次の三種類に分ける。

1. 現在のままでは検証できないもの。
2. 検証できる形へ変換できるもの。
3. 検証しなくてよいもの。

重要なのは、検証不能な主張を捨てることではなく、検証可能な条件付き主張へ書き換えることである。AMTは「全てを常に解く理論」ではなく、「証拠が不足する場合に安全に保留できる住所参照理論」である。したがって、検証できない理由を明示することは弱点ではなく、理論の安全性を高める。

## 判定区分

| 区分 | 意味 | 論文での扱い |
| --- | --- | --- |
| 形式検証済み | Leanや集合論で抽象的に証明できる。 | 定理・補題として強く書ける。 |
| 経験検証可能 | GIS、郵便データ、住所データ、配送履歴、ベンチマークで測れる。 | 実験条件、データ範囲、指標、失敗条件を明記して書く。 |
| 運用監査可能 | API、ログ、証明書、鍵管理、失効、レート制限などをテスト・監査できる。 | 実装契約または監査可能性として書く。 |
| 現在のままでは検証不能 | 全世界、全時間、全データ、全配送会社、全ユーザー行動などを含む強すぎる主張。 | 有界化、条件付き化、または未検証予想へ書き換える。 |
| 検証不要 | 定義、記法、スコープ分離、設計上の非目標、章構成、命名規則など。 | 一貫性だけ確認すればよい。実世界検証は不要。 |

## 現在のままでは検証できないもの

| 主張 | 検証できない理由 | 検証可能にする方法 | 論文での安全な書き方 |
| --- | --- | --- | --- |
| 世界中の全住所を完全に解決できる。 | 住所空間は開世界で、未登録建物、災害時住所、私的施設、自然地名、行政変更が常に発生する。 | 対象国、対象時点、対象データソース、住所種別を固定し、recall@k、候補欠落率、unresolved率を測る。 | 「宣言されたデータ範囲で候補生成 coverage を測定する。範囲外は unresolved とする。」 |
| 候補生成は真の対象を必ず含む。 | Leanの `CandidateComplete` は理想性質であり、有限の現実データから大域的に証明できない。 | gold dataset を作り、候補生成器の recall を国別・地形別・用途別に測る。 | 「候補完全性を仮定する」ではなく「候補被覆証拠を要求する」と書く。 |
| 世界全ての島名・川・湖・砂漠・湿地・氷河・遺跡を認識できる。 | 地物名は多言語、別名、季節変化、境界曖昧性、未収録地物、政治的名称差を持つ。 | OSM、GeoNames、Natural Earth、国別地名台帳、UNESCO等の source-bound benchmark を作る。 | 「対応地物型を拡張し、出典ごとの coverage を検証する。」 |
| 商用住所検証APIすべてに負けない。 | 商用APIは非公開データ、配送会社連携、国別正規化ノウハウを持ち、内部真値が公開されない。 | 同一入力セット、同一国、同一用途、同一評価指標で比較する。 | 「AMTは透明性、保留、履歴、自然地理、監査可能性で差別化する。勝敗は条件別に評価する。」 |
| 公式郵便番号APIを全て導入できる。 | APIが存在しない国、登録必須、商用制限、レート制限、利用規約不明、更新頻度不明の国がある。 | 国別 source registry に availability、license、allowedUse、freshness、rateLimit を記録する。 | 「公式ソースを優先し、利用可能性と利用条件を国別に管理する。」 |
| ZK住所証明は完全に安全である。 | envelope実装だけでは回路のsoundness、zero-knowledge、witness leakage、trusted setup安全性を証明できない。 | Circom/Noir/Rust等で回路化し、domain separation、nullifier、失効、鮮度、匿名集合を監査する。 | 「ZK-ready model」「proof envelope」と書き、完全ZKは別論文・別監査にする。 |
| AGID/AOIDはオープンソースにすれば安全である。 | OSS化は透明性を上げるが、鍵管理、QR境界、個人情報混入、API権限、同期、失効の安全性は別問題。 | threat model、secret scan、privacy boundary test、QR payload validation、API conformance test を継続する。 | 「公開可能な境界を定義し、AOID私的情報は公開データパックから分離する。」 |
| DAOで住所ルールを管理できる。 | 住所制度は行政・郵便・法律に依存し、DAOが正式性を代替できるとは限らない。 | DAOを翻訳規則、提案、監査、差分管理の補助機構に限定して評価する。 | 「DAOは正式な住所権威ではなく、補助的なガバナンス・監査層である。」 |
| Address Reputationで住所の真偽を決められる。 | 配送履歴は操作可能で、都市部偏り、差別、攻撃、返品理由の混同がある。 | 成功率、返品率、異常検知、ソース権威、バイアス監査を分けて測る。 | 「評判は到達可能性の証拠であり、住所真理そのものではない。」 |
| 住所エントロピーは都市化で必ず増える。 | 都市化以外に制度設計、グリッド、郵便コード、建物番号、住所短縮方式が影響する。 | 候補数、必要粒度、識別ビット、住所長を地域別に測る。 | 「識別に必要な情報量は、対象数と要求粒度が増えるほど増えやすい。」 |

## 検証可能に変換すべきもの

| 対象 | 検証方法 | 指標 | 成功時に書けること | 失敗時の書き換え |
| --- | --- | --- | --- | --- |
| 多言語検索 | 現地名、英語名、ローマ字、旧名、別名、非ラテン文字の検索benchmarkを作る。 | recall@k、正規化衝突率、誤結合率。 | 多言語展開は候補生成のrecallを改善する。 | 多言語展開は同一性保存ではなく、候補展開層に限定する。 |
| 自然地理検索 | 川、滝、湖、島、山、砂漠、湿地、氷河、洞窟、谷、遺跡を出典別に照合する。 | coverage、geometry validity、feature type precision。 | AMTは郵便住所外の参照対象を扱える。 | source-bound referent として扱い、未対応地物は unresolved にする。 |
| 住所タブ品質 | 都市、田舎、島、山地、砂漠、極地で品質スコアを校正する。 | false positive、false negative、再検証率。 | 品質スコアは内部判定に使える。 | スコアは真理ではなく、再検証や注意表示の信号に限定する。 |
| PID発行ゲート | 候補、クラスタ、unresolved、履歴、品質、鮮度、リスクをreplayする。 | 誤発行率、保留率、監査再現率。 | PIDは宣言された手続き的ゲートを通過したことを監査できる。 | ゲート通過は実世界真理ではなく、手続き的許容性である。 |
| 履歴グラフ | 行政変更、分筆、統合、改称、建物解体、災害住所の履歴を集める。 | split/merge表現率、後継リンク精度。 | 単一関数より履歴グラフが適切である。 | 記録欠落時は推定リンクとして扱い、証明済み履歴とは分ける。 |
| 住所検証エンジン | 国別gold dataset、郵便番号、行政階層、建物名、自然地物を照合する。 | country pass rate、postal match、admin match、deliverability proxy。 | 国別・用途別に検証能力を示せる。 | 商用API置換ではなく、透明な検証・監査層として書く。 |
| AGID/AOID通信・登録・監査 | API tests、OpenAPI conformance、QR payload tests、privacy testsを実行する。 | private field rejection、audit envelope completeness、replay resistance。 | 公開AGIDと私的AOIDの境界を実装で守れる。 | 境界違反がある場合、AOIDを中央公開データから完全分離する。 |
| ZK proof bundle | proof type、scope、challenge、audience、nullifier、freshness rootを検査する。 | replay拒否率、scope混同拒否率、互換性エラー率。 | proof同士の衝突や誤再利用を抑制できる。 | ZK機能は限定的なproof envelopeとして扱う。 |

## 検証しなくていいもの

| 項目 | 検証しなくていい理由 | 必要な確認 |
| --- | --- | --- |
| AMTの用語定義 | 定義は真偽の対象ではなく、理論内の約束である。 | 用語が全章で一貫しているか。 |
| \(X_t\)、\(Y_t\)、\(S\)、\(\mathcal{O}(X_t)\) などの記号導入 | 記号はモデルの型宣言であり、実世界データで検証するものではない。 | 型が本文、Lean、図式で矛盾しないか。 |
| AGID/AOIDをAMT本体から分離する方針 | スコープ設計の判断であり、経験的真偽ではない。 | 論文間で責務が重複しないか。 |
| ZK住所証明を別論文に分ける方針 | 暗号証明はAMTの応用であり、本体理論と分離した方が過剰主張を避けられる。 | 本体論文でZKを完成済みと書かないこと。 |
| `ambiguous`、`unresolved`、`rejected` の状態名 | 状態名そのものは設計上の命名である。 | 状態遷移とPID非発行規則がテストされているか。 |
| 章構成やPDF化 | 学術的品質には関係するが、数学的真偽ではない。 | 読みやすさ、重複、漏れ、参照整合性。 |
| 比較表の分類軸 | 分類軸は説明のための道具であり、単独で真偽を証明する対象ではない。 | 実在サービスや標準を誤説明していないか。 |
| 「本稿の範囲外」とする非目標 | 非目標は設計境界であり、検証対象ではない。 | 範囲外にしたものを本文で暗黙に主張していないか。 |

## 検証しない方がよいもの

一部の主張は、検証しなくてよいだけでなく、検証対象にしようとすると危険である。

| 項目 | 検証しない方がよい理由 | 代替 |
| --- | --- | --- |
| 個人の実住所と配送履歴を広く集める検証 | プライバシーリスクが大きい。 | 公開データ、合成データ、匿名化された集計、同意済みサンプルを使う。 |
| AOID秘密鍵やcredential本文を使う検証 | 秘密情報が漏れる。 | commitment、test vector、dummy credential、redacted envelope を使う。 |
| 商用APIの非公開レスポンスを再配布する検証 | 利用規約違反やライセンス問題がある。 | 結果は集計指標のみ公開し、レスポンス本文は保存しない。 |
| 住所評判で個人や地域をランク付けする検証 | 差別・偏り・信用毀損リスクがある。 | 住所真偽ではなく、運用上の到達可能性リスクとして扱う。 |
| 小さな集落で細かすぎるZK居住証明を公開する検証 | 匿名集合が小さく、住所秘匿にならない可能性がある。 | 最小匿名集合を満たす粒度まで粗くする。 |

## Leanで検証済みだが実世界へは直接移せないもの

| Leanで言えること | 直接は言えないこと | 実世界へ移す条件 |
| --- | --- | --- |
| 非単射観測では完全解決器は存在しない。 | どの国のどの住所入力が衝突するか。 | 実データで衝突率を測る。 |
| 候補に真の実体がなければ解決できない。 | 候補生成器が世界全体で完全であること。 | source coverage benchmark を作る。 |
| `ambiguous` / `unresolved` / `rejected` は非発行状態である。 | UIやAPIが絶対に誤発行しないこと。 | 実装テストと監査ログで確認する。 |
| 分割は単一値関数では表せない。 | 実際の行政変更履歴が完全に記録されていること。 | 履歴ソースと欠落ポリシーを定義する。 |
| 公開述語が粗ければ秘密値を識別できない場合がある。 | そのZK proofが暗号的に安全であること。 | 回路実装、匿名集合、外部監査が必要。 |

## 論文に入れる短い文案

```text
本稿では、形式検証可能な主張、経験的に検証すべき主張、運用監査で確認すべき主張、検証対象ではない定義的主張を明示的に分離する。Leanは非単射観測、候補欠落、非発行状態、履歴分割のような抽象的性質を検証できるが、実世界データの完全性、郵便APIの鮮度、自然地理データの全世界coverage、商用住所検証APIとの優劣、暗号回路の安全性を自動的には証明しない。したがって、AMTの経験的主張は、対象国、対象時点、出典、評価指標、失敗時の書き換え規則を明示した検証単位として扱う。
```

## 結論

検証できないものの多くは、理論が弱いから検証できないのではなく、主張が無条件・全世界・全時間・全データを含んでいるため検証不能になっている。

AMTでは、これらを次のように処理する。

- 数学的に証明できるものは Lean または集合論で証明する。
- 実世界データに関するものは、対象範囲を固定して benchmark に落とす。
- セキュリティやZKに関するものは、proof envelope と本物の回路監査を分ける。
- 定義、記号、スコープ分離、非目標は検証対象にしない。
- プライバシーや利用規約上危険な検証は、合成データ、匿名集計、commitment、redacted envelope に置き換える。

この整理を論文に入れると、AMTは「検証不能な巨大主張」ではなく、「証明・実験・監査・保留を分離した安全な研究プログラム」として書ける。
